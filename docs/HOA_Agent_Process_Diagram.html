4 Testing Types

Each testing type validates a different dimension of the carrier integration. The agent generates distinct request bodies per type (identified by file prefix) and applies type-specific validation logic.

1. Relevancy Testing

Purpose: Verify that required fields are correctly configured in the API and their available values match the requirement document.

Field Relevancy: Remove a mandatory field → API should error + list available values → compare with doc

Conditional: Set parent to triggering value, remove child → API should require child

rel_001   crel_001

2. Mapping Testing

Purpose: Verify that Bolt field values map correctly to carrier XML values at the expected XPaths.

Uses combinatorial packing — each scenario varies multiple independent fields from ALL blinds simultaneously.

map_001

3. Result Page Testing

Purpose: Verify that coverages and details display correctly in the API response.

Varies coverage/policy fields and checks response XML for correct display values and premium calculation.

res_001

4. UUD & Defaults

Purpose: Verify ineligibility rules trigger decline and all 20 hardcoded defaults are present in carrier XML.

UUD: Send ineligible values (Log, EFIS, 3.5, 4) → verify API declines

Defaults: Verify all 20 defaults at correct XPaths

uud_001   def_001

Agent Optimizations

Before Optimization

  • One field changed per request body
  • Organized blind-by-blind
  • Test data regenerated every time
  • All field values tested exhaustively
  • Results overwrite previous runs
  • Only mapping verification
  • Raw unformatted carrier XML
  • No bug evidence collection
  • No progress visibility

After Optimization

  • Multiple fields packed per request (combinatorial)
  • Fields from ALL blinds in one scenario
  • Smart reuse: skip / incremental update / fresh
  • Low-priority fields sampled; 15+ threshold asks user
  • Timestamped run folders preserve history
  • 4 testing types: Relevancy, Mapping, Result Page, UUD
  • Pretty-printed formatted XML alongside raw
  • 3 evidence files per FAIL for all test types
  • Granular progress: 5/25 completed

Key Features

🎯
Focused Testing Mode
User describes specific changes (e.g., "roof type mapping changed") → agent tests ONLY those fields with every possible value, conditional branch, and edge case.
📦
Combinatorial Packing
Independent fields from ALL blinds packed into each request body. Scenarios = MAX(value counts) not SUM. Reduces 20+ scenarios to 5-10.
♻
Smart Test Data Reuse
Skips generation when requirement doc unchanged. Incrementally updates when only a few things changed. Reports exact diff to user.
🔍
4 Testing Types
Relevancy (required fields + values), Mapping (field-to-XML), Result Page (coverage display), UUD & Defaults (ineligibility + hardcoded values).
🐛
Bug Evidence Collection
3 evidence files per FAIL: requirement doc reference, carrier request, carrier response. Linked from dashboard for instant download. Covers ALL test types.
📊
Tabbed HTML Dashboard
5 main tabs with sub-tabs. CSS-only (no JavaScript). Color-coded pass/fail. Evidence download links. Per-type pass rates and KPIs.
📁
Timestamped Run Folders
Every run creates {STATE}_{YYYY-MM-DD}_{HH-MM-SS}/ folder. Preserves complete history. Includes run-metadata.json with configuration.
📝
TEST_CASE_GUIDE.md
Plain-English test case descriptions grouped by testing area. Coverage matrix shows which values are tested in which scenario. Anyone can understand.
📄
Formatted Carrier XML
Raw carrier request/response from framework saved alongside properly indented formatted versions. Human-readable without losing originals.
⏳
Granular Progress
During execution: "5/25 scenarios completed". During validation: "12/25 validated". Real-time visibility into pipeline progress.
📉
Low-Priority Sampling
Occupation, industry, carrier lists: sample 2-3 values only. Fields with 15+ values: ask user first. Prevents scenario explosion.
🔄
Request Body Reuse
Checks for existing request bodies before generating. Asks user: "Reuse from previous run or generate fresh?" Saves time on repeated runs.
